This page documents which features are available in each API version. Use it to assess migration readiness or to decide which API to use for a new project.

<Tip>
  New projects should use **v2**. Features marked as v2-only are not available in v1 and have no v1 equivalent.
</Tip>

## Vehicles

| Feature | v1 | v2 |
|---------|----|----|
| `Drone` | Yes | Yes |
| `Rover` | Yes | Yes |
| `DummyVehicle` | Yes | Yes |
| `MockVehicle` (in-memory test double) | No | Yes |

## Runners

| Feature | v1 | v2 |
|---------|----|----|
| `BasicRunner` | Yes | Yes |
| `StateMachine` | Yes | Yes |
| `ZmqStateMachine` | Yes | Yes |
| Config dataclass (`BasicRunnerConfig`, `StateMachineConfig`) | No | Yes |

| `SafetyCheckerClient` / `SafetyCheckerServer` | Yes | Yes |

## Commands and control

| Feature | v1 | v2 |
|---------|----|----|
| Blocking `goto_coordinates` | Yes | Yes |
| Non-blocking goto (`VehicleTask`) | No | Yes |
| `set_velocity` with duration | Yes | Yes |
| Connection | Blocking | Async |

## v2-only features

The following features are only available in v2 and have no v1 equivalent:

- **`VehicleTask`** — issue `goto_coordinates` and other commands without blocking the calling coroutine. Poll progress, cancel in flight, or `await` completion.
- **`@at_init` in v2** — while `@at_init` exists in v1, v2 adds lifecycle guarantees around initialization ordering.
- **Command validation** — `can_takeoff(altitude)`, `can_goto(coordinate)`, and `can_land()` query the safety checker before a command is issued. Returns `(bool, str)` so scripts can handle rejections gracefully.
- **`vehicle.safety` attribute** — the attached `SafetyCheckerClient` is accessible at runtime via `vehicle.safety`.
- **Config dataclasses** — `BasicRunnerConfig` and `StateMachineConfig` allow runner configuration to be expressed as a class attribute instead of decorators.
- **Typed exceptions** — `VehicleCommandError`, `SafetyCheckError`, and related exception types replace generic `Exception` and `RuntimeError`.
- **`MockVehicle`** — an in-memory vehicle implementation for unit testing scripts without hardware or SITL.
- **Async connection** — `Drone.connect()` and `Rover.connect()` are `async` methods, enabling connection as part of an async workflow.
